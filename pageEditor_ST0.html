Feedback Page

Technical Description

The Feedback page is built using a main container divided into three distinct segments: a programme directory, a participant reviews grid, and a feedback submission form. The programme directory utilizes a CSS Grid (.prog-grid) with responsive auto-fill and minmax(240px, 1fr) settings, housed within a scrollable container (.prog-scroll) that has a max-height: 420px and overflow-y: auto. This ensures that even as more programmes are added, the overall page height remains manageable. The Feedback form uses semantic fieldset and legend tags to group personal data and feedback details, while a datalist provides native, accessible search suggestions for the interests input field.

Accessibility

Accessibility and user feedback are addressed through robust JavaScript features. A live character counter for the message textarea provides immediate visual feedback, implemented with an input event listener that adjusts text colour as users approach the 300-character limit. Custom validation logic on form submission uses regular expressions to verify email formats and minimum message lengths, toggling the visibility of specific error messages (.error-msg) which are coupled with input highlight styling (.input-error). Upon successful validation, the form is hidden and a confirmation message with role="alert" is displayed, ensuring immediate notification for screen reader users that their submission was processed.

Team Page

Technical Description

The Team page is built on the basis of the CSS Grid created with grid-template-columns: repeat(auto-fit, minmax(240px, 320px)) and justify-content: center. This causes the cards to react to the screen width in a natural manner without the need to provide explicit media queries of columns wrapping automatically as the space gets smaller.

Every member of the team is identified by "a .member-block div" in the form of an interactive card. Position: relative, border-radius, overflow: hidden, and a CSS transition on transform, box-shadow, and border-color are utilized in the card to create the effect of a lift-and-highlight on hover (translateY(-6px)).

Member profile images are contained in .member-thumb - a fixed height flex box (height: 200px) with a gradient fallback background. Object-fit: cover is in images to occupy the thumbnail without distortion of the image it represents, irrespective of the aspect ratio of the source image.

By default, the expandable bio section (.member-details) is not revealed by max-height: 0; overflow: hidden and only shown on:hover and :focus-within by animating max-height to 200px. This is a CSS-based accordion system that does not need any JavaScript yet has a good, animated opening of the contents.

Accessibility

All member-blocks have tabindex=0 therefore keyboard users can navigate and focus on cards without using the mouse. The :focus-within pseudo-class replicates the hover behaviour, i.e., keyboard focus animations (expands) are the same as what happen when the mouse hovers - allowing full access to the keyboard.

role="list" on .team-grid and role="listitem" on the respective cards replaces lost list semantics, as using div elements loses the default list role. aria-label of every card gives a descriptive name that is announced by screen readers (e.g. Team member: Student 1).

.member-thumb aria-hidden=true allows ornamental images and gradient spaces to not fill the screen reader display. The presence of aria-live=polite on the element of the member-details indicates to assistive technologies that the content has changed dynamically when a card on the screen is focused or hovered, and therefore the information became visible, and the users of the screen readers are informed of that.

The Google Fonts link tags include rel=preconnect to connect to Google's font servers early, which reduces the time it takes for the fonts to load and avoids a flash of unstyled text on slower connections.

Content Page

Technical Description

The CSS Layout of the Content page is based on the two-column grid (.content-layout) configured with grid-template-columns: 220px 1fr wherein the former column contains a sticky sidebar navigation and the latter column contains the main article. The align-items: start that is selected means that both columns are positioned at the same height as opposed to being equal without extending. One of the breakpoints, a breakpoint of 720 pixels, takes the grid to a single column and covers the sidebar on smaller screens to keep the page clean on the mobile devices.

The sidebar (.side-nav) has position: sticky and top: 80px so that it is visible as the article is scrolled by the user. Both navigational links are defined as border-left: 3px solid transparent and have a transition on the color and border-color, which causes a slight highlight of the left border to show on hover and allows the user to visually know which part the link will take them to.

The article is mainly contained under a semantic tag of article and breaks down into section elements with unique id and which are linked to by side bar anchor. Border-bottom: 2px solid var(--border) has been used in each of the h2 headings to visually divide sections. The images within the content are represented by the class content-img with the following properties: object-fit: cover, max-height: 280px, and border-radius: 10px in order to be represented in the case of non-distortion, irrespective of the size of the original image.

To ensure that it is always available and does not discontinue the flow of reading, there is a permanently placed button at the bottom: 2rem; right: 2rem with the help of position: fixed. It is connected to a hidden anchor which is called page-top that is located at the very top of the body with a top: -80px offset in order to consider the height of the sticky header.

Accessibility

On the aside element, the aria-label="Page sections" has been added to the sidebar so that screen readers can identify it as a navigation landmark that is independent of the core navigation. The primary nav in the header also has its own aria-label="Main navigation" to make it clear that there are two distinct areas of navigation.

The "back to top" link includes an aria-label="Go to top of page" since the visible textual information is only an arrow (↑). Without the label, this would make no sense to a screen reader. Any images in the content have descriptive alt text detailing the subject of the image and its importance to that section, ensuring users of screen readers are provided with the same context as those viewing the images visually.

Lastly, Google Fonts uses the rel="preconnect" attribute on the <link> elements to preload connections. This ensures font load latency is reduced to a bare minimum and a flash of unstyled text is not shown.

User Profile Page

Technical Description

The User Profile page utilizes a two-column CSS Grid layout (.profile-layout) defined with grid-template-columns: 1fr 1.8fr. This layout provides a narrower left column for the control panel and a wider right column for the profile output. At a max-width of 680px, a media query activates to redefine the grid as a single column (1fr), ensuring the layout stacks naturally without horizontal scrolling on mobile devices.

The interface relies heavily on JavaScript to create an interactive, multi-step progressive profiling experience. Data is collected sequentially using the native prompt() function rather than standard form inputs. A JavaScript data object (profileData) temporarily stores these inputs. Once a step is completed, DOM manipulation updates the right panel spans (e.g., #out-name) with the captured data, and the corresponding .profile-section div is toggled from display: none to display: block.

Visual feedback is handled dynamically via CSS classes and JavaScript. When a step is finished, JS adds the .done class to the step button, changing its background colour. Simultaneously, a smooth CSS keyframe animation (@keyframes fadeIn) applies a translateY lift and opacity fade to reveal the newly populated section. The progress bar's width is recalculated in JS based on completed steps and animated using a CSS transition: width 0.5s.

Accessibility

The page uses semantic HTML5 landmarks, including aside and main, to provide clear document structure for screen readers. The aside element is labelled with aria-label="Profile steps" to describe its contents to assistive technologies, while the main output area uses aria-label="Your profile".

The profile output container includes aria-live="polite", which ensures screen reader users are automatically notified when new profile data is injected into the DOM after completing a step, without aggressively interrupting their current reading flow. When all three steps are finished, a completion message appears utilizing role="status" to announce the success state.

For the custom progress bar, standard ARIA progress attributes are used. The wrapper div utilizes role="progressbar", alongside aria-valuemin="0", aria-valuemax="100", and a dynamically updating aria-valuenow to accurately communicate the percentage of completion to visually impaired users as they progress through the steps.

Sitemap Page

Technical Description

The Sitemap page abandons the traditional approach of using a static image or nested HTML lists in favour of a fully interactive, inline Scalable Vector Graphic (SVG). The map is enclosed in an <svg> element that uses a viewBox="0 0 860 560" attribute combined with width="100%". This ensures the entire diagram scales responsively down to mobile screens without breaking the layout or losing layout proportions. The diagram is wrapped in a container (.sitemap-wrap) with overflow-x: auto to provide secure horizontal scrolling on very narrow devices if the scaling becomes too small to read.

Every node on the sitemap acts as a functional hyperlink. The nodes are structured using SVG <a> (anchor) tags containing <rect> shapes for the card backgrounds and <text> elements for the labels. The connector lines are drawn using explicit <line> coordinates with stroke and stroke-dasharray styling to visually distinguish hierarchy levels and content groupings.

Interactivity is managed entirely via CSS. When a user hovers or focuses on a node (.sm-node:hover rect), CSS transitions smoothly animate the fill and stroke properties to highlight the selection. A crucial UX detail is the application of pointer-events: none on the SVG text elements. This prevents the hover effect from rapidly flickering on and off when the mouse cursor passes over the text characters inside the rectangle.

Accessibility

Building a complex diagram in SVG presents accessibility challenges, which were addressed by making the SVG intrinsically accessible. The root <svg> element is given role="img" and a descriptive aria-label so screen readers recognize it as a single coherent diagram rather than a random collection of lines and shapes.

Every interactive node (<a class="sm-node">) is given tabindex="0" to guarantee keyboard focusability. Because SVG anchor tags do not always announce their inner content reliably across all screen readers, an explicit aria-label describes the exact destination of the link (e.g. "Content page: Sustainable Fishing"). Additionally, a native SVG <title> element is nested inside each node to provide a standard tooltip on mouse hover.

To support keyboard navigation visibility, a custom :focus state applies a high-contrast dashed outline (outline: 2px dashed var(--secondary)) with an outline-offset, ensuring keyboard users can clearly see which node they are currently tabbed onto.
