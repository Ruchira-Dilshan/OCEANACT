Feedback Page

Technical Description

The Feedback page is constructed with the help of primary container that is subdivided into three separate parts: programme directory, a grid of reviews, and a form of feedback submission. The programme directory has a CSS Grid (.prog-grid), responsive settings of auto-fill and minmax(240px, 1fr) and placed in a scrollable container (.prog-scroll), with max-height: 420px and overflow-y: auto. This makes sure that with the addition of more programmes the overall page height can be maintained under control. Feedback form consists of semantic fieldset and legend tags to categorize personal information and feedback content, and datalist to offer native, accessible search recommendations to the field of interests.

Accessibility

The availability and response are covered with powerful JavaScript functionality. An immediate visual feedback to show the amount of characters typed into a message textarea uses a live character counter, which was executed using an input event listener which modifies the text colour as the user nears the 300-character limit. Form submission validation is done via regular expressions to validate email format and minimum length of message and show or hide certain error messages (.error-msg) that are coupled with input highlighting (.input-error). When the form is successfully validated, the form is replaced with a confirmation message having role = alert and this will immediately notify the users of the screen reader that their form has been submitted.

Team Page

Technical Description

The Team page is built on the basis of the CSS Grid created with grid-template-columns: repeat(auto-fit, minmax(240px, 320px)) and justify-content: center. This causes the cards to react to the screen width in a natural manner without the need to provide explicit media queries of columns wrapping automatically as the space gets smaller.

Every member of the team is identified by "a .member-block div" in the form of an interactive card. Position: relative, border-radius, overflow: hidden, and a CSS transition on transform, box-shadow, and border-color are utilized in the card to create the effect of a lift-and-highlight on hover (translateY(-6px)).

Member profile images are contained in .member-thumb - a fixed height flex box (height: 200px) with a gradient fallback background. Object-fit: cover is in images to occupy the thumbnail without distortion of the image it represents, irrespective of the aspect ratio of the source image.

By default, the expandable bio section (.member-details) is not revealed by max-height: 0; overflow: hidden and only shown on:hover and :focus-within by animating max-height to 200px. This is a CSS-based accordion system that does not need any JavaScript yet has a good, animated opening of the contents.

Accessibility

List lost semantics are replaced by role of .team-grid and roleitem of the cards, and the default list role of a div element is lost, which is why aria-label of each card provides a descriptive name read out by screen readers (e.g. Team member: Student 1).

member-thumb aria-hiding=true will not make decorative images and spaces of gradient fill the screen reader view. The use of aria-live=polite in the element of the member-details demonstrates to the assistive technologies that the content has dynamically changed when one card on the screen is in focus or hovered and, thus, the information became visible, and it is notified to the users of the screen readers.

The tags with rel=preconnect are to preconnect with Google font servers, which decreases the time spent on loading the fonts and eliminates the flash of unstyled text in slower connections.

Content Page

Technical Description

The CSS Layout of the Content page is based on the two-column grid (.content-layout) configured with grid-template-columns: 220px 1fr wherein the former column contains a sticky sidebar navigation and the latter column contains the main article. The align-items: start that is selected means that both columns are positioned at the same height as opposed to being equal without extending. One of the breakpoints, a breakpoint of 720 pixels, takes the grid to a single column and covers the sidebar on smaller screens to keep the page clean on the mobile devices.

The sidebar (.side-nav) has position: sticky and top: 80px so that it is visible as the article is scrolled by the user. Both navigational links are defined as border-left: 3px solid transparent and have a transition on the color and border-color, which causes a slight highlight of the left border to show on hover and allows the user to visually know which part the link will take them to.

The article is mainly contained under a semantic tag of article and breaks down into section elements with unique id and which are linked to by side bar anchor. Border-bottom: 2px solid var(--border) has been used in each of the h2 headings to visually divide sections. The images within the content are represented by the class content-img with the following properties: object-fit: cover, max-height: 280px, and border-radius: 10px in order to be represented in the case of non-distortion, irrespective of the size of the original image.

To ensure that it is always available and does not discontinue the flow of reading, there is a permanently placed button at the bottom: 2rem; right: 2rem with the help of position: fixed. It is connected to a hidden anchor which is called page-top that is located at the very top of the body with a top: -80px offset in order to consider the height of the sticky header.

Accessibility

The aria-label=Page sections has been added on the aside element on the sidebar to ensure that the screen readers identify it as a landmark on the navigation which is not bound with the main navigation. The primary nav within the header also possesses an aria-label of its own, which is "Main navigation" to ensure that there is a differentiation that there are two different areas of navigation.

The "back to top" link does contain an aria-label of Go to top of page as the visible textual information is merely an arrow ( ↑). This is nonsense with no label to a screen reader. The content includes any images and they have descriptive alt text which explains what the image is all about and how it relates to the content in that section and thus a user of a screen reader will have the same level of understanding as a person who is viewing the images.

Finally, Google Fonts includes the rel=preconnect attribute of the elements to preload connections. This makes sure that font load latency is minimized to the lowest possible and there is no flash of un-styled text displayed.

User Profile Page

Technical Description

This User Profile page is based on the CSS Grid layout (.profile-layout) with grid-template-columns: 1fr 1.8fr. This design gives the control panel a slimmer left column and a broader right column to the profile output. Simultaneously, when the maximum width is reached (680px), a media query will trigger the rearrangement to a single column (1fr), and the layout will be stacked by default without horizontal scrolling on a mobile.

The interface greatly depends on JavaScript to produce a multi-step, interactive progressive profiling. The native prompt() function is used to gather data in a sequential manner instead of using the normal form data. These inputs are temporarily stored in a JavaScript data object (profileData). After each step has been done, DOM manipulation is used to change the spans in the right panel ( e.g. #out-name ) to the data that was captured, and the associated .profile-section div is changed to display: none display: block.

The display of visual feedback is done dynamically through CSS classes and JavaScript. Upon completion of a step,JS attaches the.done class to the step button altering its background colour. At the same time, there is a seamless CSS keyframe transition of a fade in ([@keyframes fadein]). It gives a lift of translateY and a decrease in the amount of opacitance to unveil the newly filled area. The width of the progress bar is recalculated in JS on the basis of completed steps and is animated with the help of a CSS transition: width 0.5s.

Accessibility

TSemantic HTML5 landmarks such as aside, main, have been used on the page to create easy document structure by the screen readers. The aside element is marked with aria-label="Profile steps" to tell the contents of the piece to assistive technologies whereas the output area is marked with aria-label="Your profile" as the main output.

TThe profile output container contains aria-live=polite which guarantees that users of a screen reader are automatically informed that new profile data is injected into the DOM once a step has been completed but it does not forcefully interrupt their ongoing reading process. Upon completion of the three steps, a completion message is displayed using role= status to declare the success status.

The custom progress bar will be using the standard ARIA progress attributes. This wrapper div has role as progressbar and aria-valuemin = 0, aria-valuemax = 100 as well as dynamically set aria-valuenow so that the percentage of completion is well portrayed to the visually impaired users as they continue with the steps.

Sitemap Page

Technical Description

The Sitemap page also does not conform to the old pattern of using a static image or nested HTML lists in favour of a fully interactive, inline Scalable Vector Graphic (SVG). The map itself is contained in an element with a viewBox attribute of 0 0 860 560 and a width attribute of 100%. This will make the whole diagram scale down to the screen of mobile devices in a responsive way without any reduction in the layout or proportions in the layout. The diagram is encased in an element (.sitemap-wrap) with an overflow-x auto in case it has to scroll horizontally safely on very skinny devices in case the scaling is too small to read.

Each item of the sitemap is a functional hyperlink. The nodes are drawn with SVG with tags inside of the nodes of the form of to make the background of the cards and with tags to form the labels. To make the hierarchy levels and content groupings visually distinct the connector lines are drawn with explicit line coordinates explicitly styled with a stroke and stroke-dasharray.

The interactivity is controlled through CSS. On hovering or emphasizing a node (.sm-node:hover rect), CSS transitions are used to automatically change the fill and stroke properties that are used to emphasize the selection. The use of pointer-events: none on SVG texts is a very important UX detail. This will avoid the rapid flickering on and off appearance of the hover effect in cases where the mouse cursor floats across the characters of the text within the rectangle.

Accessibility

Creating a complicated diagram in SVG poses some challenges to accessibility and by making the SVG intrinsically accessible these challenges were overcome. The root element is assigned the role of img and a descriptive aria-label to ensure that a screen reader reveals that it is a single coherent diagram and not a haphazard set of lines and shapes.

In order to make keyboard navigation visible, the custom focus state gives high-contrast dashed outline (outline: 2px dashed var(--secondary)) with outline-offset, so that the keyboard users can actually see what node they are currently tabbed into.

Challenges and Lessons Learned

The interactive SVG sitemap implementation was one of the major technical challenges that were faced. In particular, there was a problem in the flickering of the hover transitions during the movement of the cursor between text labels in the nodes. This was solved by applying the CSS pointer-events: none property on all of the SVG text elements, so that it was only the node rectangles that generated the hover state leading to a far much smoother user experience.

The other issue was to ensure consistency of layout across devices, but without repetitive media queries. With the Team and Feedback pages designed using CSS grid and its auto-fit and minmax() features, I designed a fluid layout that wraps itself. But to be sure that the object-fit: cover property was working properly with images of different aspect ratios within these grids, it took some experimentation with fixed-height containers to avoid distortion of the image.

It also took particular care to implement JavaScript in establishing real time form validation and feedback on the Feedback page. I was able to write the live character counter which changes color when the user is approaching the limit. Among the lessons learnt here was that instead of using the keyup event, we should use the input event because it is the only event that would capture all types of text entries such as pasting and this makes the validation system more robust.

Lastly, this project strengthened the importance of semantic HTML and creating accessibility-friendly designs. aria-live in the dynamic profile and role=alert in the form submission made me understand the role of making components interactive to be really useful to the users of the screen reader. Also, the shift to a standardized images/ folder structure meant that there was a need to have good management of the assets so that paths would not be broken in a multi-page project.
